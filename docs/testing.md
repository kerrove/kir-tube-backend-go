# Тестирование

В проекте три уровня тестов, все на `testing` + `stretchr/testify`.

## Уровни

| Уровень | Файлы | Что проверяет | Нужна инфраструктура |
|---------|-------|---------------|----------------------|
| **Юнит** | `*_test.go` рядом с сервисами/пакетами (`service_test.go`, `pkg/*/‌*_test.go`) | Бизнес-логика на моках/фейках | Нет |
| **Интеграционные** | `internal/*/repository_int_test.go` | Репозитории против реальной PostgreSQL | PostgreSQL (тестовая БД) |
| **E2E** | `internal/app/e2e_test.go` | Всё приложение через реальный HTTP | PostgreSQL + фейковое хранилище |

## Запуск

```sh
go test ./...                 # все тесты
go test -short ./...          # только юнит (БД-тесты пропускаются)
go test ./internal/video/...  # конкретный пакет
go test -run TestName ./...   # конкретный тест
go test -race -cover ./...    # гонки + покрытие
```

Тесты, требующие БД, **пропускаются** (`t.Skip`, не падают) при:
- флаге `-short`;
- недоступной тестовой базе.

Поэтому «чистый» юнит-прогон не требует никакой инфраструктуры.

## Инфраструктура тестов (`pkg/testutil`)

Пакет намеренно **не импортирует** `internal/app` — чтобы white-box тесты могли
им пользоваться без цикла импортов. E2E-харнесс собирает приложение через
`app.Assemble` из собственного тест-файла.

- **`RequireDB(t)`** — соединение с тестовой БД: применяет `AutoMigrate`,
  очищает все таблицы до и после теста. Пропускает тест при `-short` или
  недоступной базе.
- **`TruncateAll(t, db)`** — `TRUNCATE ... RESTART IDENTITY CASCADE` по всем
  таблицам public-схемы: каждый тест стартует с чистого листа.
- **`TestConfig()`** — конфиг для тестов (тестовый секрет, небезопасные cookie,
  фиксированный `CLIENT_URL`, DSN тестовой БД).
- **`IssueToken(t, userID)`** — выпускает валидный access-токен (1 час), чтобы
  дёргать защищённые эндпоинты.
- **`FakeStorage`** — in-memory реализация `media.ObjectStorage`: медиа-флоу
  тестируются без MinIO. Есть хелперы `Has(key)` и `Keys()`.
- **Фикстуры** (`fixtures.go`) — хелперы для вставки строк.

### Изоляция БД-тестов между процессами

`go test` запускает тест-бинарники разных пакетов **параллельно** против одной
общей тестовой БД. Чтобы они не конкурировали за `AutoMigrate` и не затирали
данные друг друга, `RequireDB` держит **session-level advisory lock** PostgreSQL
(`pg_advisory_lock`) на выделенном соединении на всё время теста — БД-тесты
становятся взаимно исключающими глобально.

## Тестовая база данных

По умолчанию используется база `kir-tube_test` на локальном PostgreSQL из
docker-compose. DSN переопределяется переменной **`TEST_DSN`**.

Подготовка:

```sh
docker compose up db minio           # поднять зависимости
# создать тестовую БД (её нет в дефолтном образе — есть только POSTGRES_DB):
docker exec -it kir-tube-db createdb -U postgres kir-tube_test
```

> Контейнер PostgreSQL создаёт только базу из `POSTGRES_DB` (`kir-tube`). Базу
> `kir-tube_test` под интеграционные и e2e тесты нужно создать отдельно (команда
> выше), иначе такие тесты будут пропущены.

## E2E-харнесс

`internal/app/e2e_test.go` собирает приложение с тестовой БД и `FakeStorage`,
поднимает `httptest.NewServer` и гоняет реальные HTTP-запросы через клиент с
`cookiejar` (чтобы работали refresh-cookie). Это самый близкий к продакшену
уровень проверки — маршрутизация, middleware, сериализация и БД вместе.
