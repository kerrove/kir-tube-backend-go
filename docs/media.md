# Медиа: загрузка и транскодирование

Медиа-модуль (`internal/media`) отвечает за приём файлов, транскодирование видео
в несколько разрешений и раздачу статики. Блобы хранятся в **MinIO** (S3).

## Абстракция хранилища

Сервис и хендлер зависят от интерфейса `ObjectStorage`, а не от клиента MinIO
напрямую (Dependency Inversion) — это позволяет подменять хранилище в тестах на
in-memory фейк:

```go
type ObjectStorage interface {
    Put(ctx, key string, r io.Reader, size int64, contentType string) error
    Get(ctx, key string) (io.ReadCloser, ObjectInfo, error)
}
```

Продакшн-реализация — `MinioStorage`. При старте она подключается к MinIO и
**создаёт бакет**, если его нет. В `app.Assemble` хранилище можно передать явно;
если передан `nil`, модуль сам строит `MinioStorage` из конфига (продакшн-путь).

**Схема ключей** совпадает с публичными URL:
- Оригинал: `<folder>/<name>`
- Вариант разрешения: `<folder>/<resolution>/<name>`

## Загрузка файла

`POST /api/upload-file?folder=<folder>` (требует авторизации), multipart-поле
формы — `file`.

Валидация (`internal/media/validation.go`):
- **Папки:** `avatars`, `banners`, `thumbnails`, `videos`. Пустая → `default`.
- **MIME-типы:** `image/jpeg`, `image/png`, `image/svg+xml`, `video/mp4`,
  `video/quicktime`.
- **Размер:** до **300 МБ** на файл.
- Лимит multipart-памяти при разборе — 64 МБ.

Порядок обработки (`MediaService.SaveMedia`):
1. Генерируется уникальное имя файла, формируется ключ `<folder>/<name>`.
2. Оригинал загружается в MinIO.
3. **Не видео** → сразу возвращается `{ url, name }`.
4. **Видео** → оригинал складывается во временный файл (ffmpeg/ffprobe работают с
   путями), через `ffprobe` определяется разрешение, вычисляется `maxResolution`,
   статус ставится в «в очереди», и запускается **фоновая горутина**
   транскодирования. Ответ возвращается сразу с `maxResolution`.

Ответ — массив `MediaResponse`:
```json
[{ "url": "/uploads/videos/abc123.mp4", "name": "abc123.mp4", "maxResolution": "1080p" }]
```

## Транскодирование видео

Выполняется в фоне (`processVideo` → `convertVideo`), требует **`ffmpeg`** и
**`ffprobe`** в `PATH`:

1. `ffprobe` читает ширину/высоту исходника.
2. Из списка `Resolutions` выбираются все, что **не превышают** исходное
   разрешение.
3. Для каждого целевого разрешения `ffmpeg` масштабирует видео во временный файл
   и загружает его в MinIO по ключу `<folder>/<resolution>/<name>`.
4. После каждого разрешения обновляется прогресс.

Метки качества (`Quality*`): `4K`, `2K`, `1080p`, `720p`, `480p`, `360p`.
`mapResolution` выбирает наибольшую метку, которой достигают размеры (иначе
`720p`).

## Статус обработки

`GET /api/upload-file/status/{fileName}` (требует авторизации):
```json
{ "fileName": "abc123.mp4", "status": 100 }
```
Значения `status`:
- `0` — в очереди или неизвестно
- `1..100` — прогресс (обновляется после каждого готового разрешения); `100` — готово
- `-1` — ошибка транскодирования

Статусы хранятся в памяти сервиса под `sync.RWMutex` (`processingStatus`), то есть
**не переживают перезапуск** и не разделяются между несколькими инстансами.

## Раздача статики

`GET /uploads/{path...}` — стримит объект из MinIO. Регистрируется на корневом
мультиплексоре (вне `/api`) и с алиасом `/api/uploads/...` для URL, сохранённых
до разделения путей.

Безопасность и поведение:
- Путь нормализуется (`path.Clean`); обход каталога (`..`) → `403`.
- Листинг каталога (`/uploads`, `/uploads/index.html`) → `403`.
- Отсутствующий объект → `404`.
- Проставляются `Content-Type` и `Content-Length` из метаданных объекта.

## Замечания по эксплуатации

- Для работы транскодирования в контейнере backend нужны `ffmpeg`/`ffprobe`.
  Текущий `Dockerfile` собирает бинарь на `scratch` — двоичных ffmpeg там нет,
  так что транскодирование в этом образе работать не будет (загрузка изображений
  и раздача — будут). Для видео потребуется базовый образ с ffmpeg.
- Статус обработки — в памяти процесса: при горизонтальном масштабировании его
  нужно вынести во внешнее хранилище.
